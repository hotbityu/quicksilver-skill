# Quicksilver Skill

## Overview

Use this skill when the user asks for Quicksilver or QS project work, including exporting or discovering project APIs. The skill is organized by sub-skill, with detailed workflow references and templates stored separately.

## Sub-Skills

### Create QS Project

Use this sub-skill when the user asks to create, scaffold, initialize, or describe a QS project.

- Read `references/create-qs-project.md` for the expected QS project structure and workflow.
- Copy template files and directories from `assets/templates/create-qs-project/<project-type>/` when creating parent projects or subprojects.
- Render template placeholders while copying files, including `<parent-project-name>` and `<qs-version>` in `build.gradle` templates.
- If `<qs-version>` is required, use the user-provided QS version first; only try the configured Maven metadata URL when the user did not provide a version.
- Treat the project name as a user-provided prefix, such as `ecp` or `mes`.
- If the user specifies where to create the QS project, create it at that location; otherwise create `<project-name>/` under the current command-line working directory.
- When the user only provides a project name, create the default skeleton: `<project-name>-build/`, `<project-name>-run/`, and `<project-name>-module-main/`.
- When creating the default parent project skeleton, include `<project-name>-build`, `<project-name>-run`, and `<project-name>-module-main` in `settings.gradle`.
- When the user asks to create a `lib` or `module` subproject under an existing QS parent project, name it `<project-name>-lib-<subproject-name>` or `<project-name>-module-<subproject-name>`.
- After creating any subproject, ensure the parent project's `settings.gradle` includes it, for example `include 'abc-lib-zip'`.
- After creating any `module` subproject, including modules created during parent project initialization, ensure `<project-name>-run/build.gradle` depends on it, for example `implementation project(':abc-module-main')`.
- When adding a unit to a project with multiple module projects, determine the exact target module project before generating SQL or Java files. If it cannot be inferred, ask which module should receive the unit.
- For unit SQL, use the target module project's `src/main/resources/QS-MODULE/data/sql/default` tree. Only module projects such as `<project-name>-module-main` or `<project-name>-module-xxx` have this QS-MODULE SQL directory. If that tree has nested directories, sort directory names lexicographically at each level and create the new SQL in the latest final directory; otherwise create it directly under `default`. Name the file by incrementing the latest SQL filename's final numeric segment, for example `quicksilver-7.1.31.21.sql` -> `quicksilver-7.1.31.22.sql`.
- When adding a unit, if `init.sql` exists under the selected module project's `src/main/resources/QS-MODULE/data/sql/default` tree, also sync the generated unit SQL into that `init.sql`.
- For unit modify/delete SQL, scan the selected module project's `src/main/resources/QS-MODULE/data/sql/default` tree for the original `insert` statement, extract the row's `FId`, and generate `update` or `delete` using `where FId='<id>'`. Add a `--<name-or-title>` comment immediately above each generated `update` or `delete`, taking the label from the original insert's `FName` or `FTitle`.
- If the user asks to add an entity unit but does not say whether Java files should be generated, ask before creating Java files. Non-entity units are Java method entry points, so generate Java by default unless the user explicitly says not to. When Java generation is requested, keep generated Java files under the selected target module project's `src/main/java`, in the directory matching the requested Java package. When Java generation is not requested, do not auto-populate `TsUnit` Java class binding fields unless the user provides class names.

### Unit

Use this sub-skill when the user asks to add, modify, delete, or generate a QS unit, including unit SQL or Java files.

- Read `references/unit.md` before generating or changing unit files.
- Use `references/unit-system-tables.md` when generating or changing dictionaries, menus, privileges, roles, text resources, or other common QS system metadata rows.
- Inspect the target QS module first and match its existing package names, DAO/service/action/home patterns, SQL location, naming style, and delete conventions.
- Copy or adapt template files from `assets/templates/unit/` only after resolving placeholders from the target project and user-provided unit details.
- Treat add, modify, and delete as separate operations: add creates the unit files, modify updates the existing unit files in place, and delete removes or disables the unit according to the target project's established convention.
- Ask for missing unit code, unit name, table name, fields, module name, base class, or delete semantics only when they cannot be inferred from the target project.

### Text Resource

Use this when the user asks to add, modify, or delete QS text resources for i18n.

- `TsTextResource` stores base i18n text resources. Frontend/backend code references `FCode`, and QS resolves `FValue` by the current language through the system i18n mechanism.
- Text resource codes normally start with `T.` for text/UI resources or `E.` for error/exception messages.
- For add SQL, require `FCode` and `FValue`, generate a fresh UUID, and use `insert into TsTextResource set FId='<uuid>', FCode='<code>', FValue='<value>';`.
- For modify SQL, require `FCode` and the new `FValue`, and generate `update TsTextResource set FValue='<value>' where FCode='<code>';`.
- For delete SQL, the user-provided target may be either `FCode` or `FValue`. Treat values matching `T.*` or `E.*` as `FCode`. For ordinary text, search existing SQL for a matching `TsTextResource.FValue`; if one matching resource is found, delete by its `FCode`. If none or multiple are found, report that and only use `where FValue='<value>'` when the user wants value-based deletion.
- Escape single quotes in SQL string values by doubling them.

### API Discovery

Use this sub-skill when the user asks to discover, analyze, inventory, document, or export APIs across QS module projects.

- Read `references/api-discovery.md` before scanning module APIs.
- For fast first-pass exports, prefer running `scripts/export_qs_api.py <project-root> [output-md]` after locating the QS parent project. If the user gives a single `*-module-*` directory, treat its parent as the QS project root when sibling modules/settings are present. Use the generated Markdown as the initial API reference, then manually inspect and refine complex handlers when service-level tracing, dynamic object shapes, or unresolved inference requires more evidence.
- Use `assets/templates/api-discovery/java/` as the API annotation, inherited API, and result contract reference, especially `Api.java`, `BaseApiImpl.java`, `EntityApiImpl.java`, `ApiAttribute.java`, `ApiDataType.java`, `ApiRequestFormat.java`, and the `ApiResult` implementations.
- Discover all module projects first, such as `<project-name>-module-main` and `<project-name>-module-xxx`. Analyze every discovered module by default; limit to one module only when the user explicitly requests it.
- Scan only each module's `api.impl` `XXXApiImpl` classes for the primary pass, such as `com.jeedsoft.quicksilver.account.api.impl.AccountApiImpl`; each method annotated with `@Api` is the handler for that API request. Do not parse every Java file up front for API discovery. Resolve additional Java classes lazily only when a confirmed handler needs them for superclass, model, or field-shape evidence.
- Include inherited `@Api` methods by walking the actual superclass chain, expanding them into concrete APIs with the concrete unit's OpenAPI route. Treat `BaseApiImpl`, `EntityApiImpl`, `BusinessEntityApiImpl`, and `TreeEntityApiImpl` as top-level/base API classes: they are contract sources only and must not be exported as standalone APIs for themselves. Use the bundled `BaseApiImpl.java` template to resolve non-entity base classes when source is missing. Current `EntityApiImpl` contains the legacy `BusinessEntityApiImpl` and `TreeEntityApiImpl` behavior; old projects may still extend those deprecated classes, so use the bundled `BusinessEntityApiImpl.java`, `TreeEntityApiImpl.java`, and `EntityApiImpl.java` templates to resolve missing legacy base classes and continue the chain to `EntityApiImpl`. Do not leave inherited APIs only as generic templates when the concrete unit can be resolved. Put inherited source and contract details or concise inherited-handler notes inside each concrete route's path-headed API section; do not create a standalone shared `EntityApiImpl` contract section. API classes that stay on `BaseApiImpl`, such as non-entity units like `MiscApiImpl`, do not inherit the entity-style `item/create/...` APIs.
- For APIs with `@Api.isFullPowers=true`, treat `@Api.isTokenRequired` as not applicable: the developer fully parses `request` and writes `response`, so do not auto-add an `Authorization` header from token metadata.
- Treat token requirement as code-first. If handler code calls `getTokenId()` on the API method parameter typed as `ApiContext`, such as `public JsonResult discard(ApiContext ac)` with `ac.getTokenId()` or `public JsonResult discard(ApiContext context)` with `context.getTokenId()`, report `是否需要 Token` as `true` and add the `Authorization` header even when `@Api(isTokenRequired=false)` is declared, because the handler needs the caller's token for business behavior. Do not treat arbitrary non-`ApiContext` `getTokenId()` calls, static `ApiContext.getTokenId()` calls, or local-variable calls as token requirements. Mark this explicitly in `基本資訊`, for example `true（處理器程式碼呼叫 ApiContext 參數的 getTokenId()；覆蓋 @Api.isTokenRequired=false）`.
- For file-upload APIs, detect `ApiContext.getFileItem()` and `ApiContext.getFileItems()` from handler code even when `@Api.input` does not declare files. Mark these APIs as `multipart/form-data`. In the generated frontend calling reference, explain that callers use `FormData`: ordinary parameters are commonly grouped into an `args` JSON string such as `formData.append("args", JSON.stringify({life: "temp"}))`, may also be appended directly, and file fields such as `file1` or `file2` follow QS upload parsing conventions. `getFileItem()` reads the first uploaded file; `getFileItems()` reads the uploaded file list.
- For very large `ApiImpl` classes, classes with many `@Api` handlers, or handlers with very large method bodies, prefer annotation-first extraction and skip deep method-body field scanning to avoid one complex class dominating export time. Ignore `@Api` blocks inside Java comments. Keep `@Api.input` and `@Api.output` fields, path, token, and response metadata.
- Keep type inference conservative across files and dependencies. Infer response/request field types from literals, local variables, common JSON/`ApiResult.put(...)` patterns, `@ApiAttribute` metadata, and exception-object text getters such as `e.getMessage()` and `e.getStackTrace()` as `string` only when `e` can be identified as an exception/throwable variable. For direct constants such as `SomeErrorCode.SYSTEM_ERROR`, resolve only lightweight `public static final String/Integer/Long/Boolean/...` declarations from source files under the currently scanned repository's module `src/main/java` trees. When a request default value or response expression references a Java constant or enum such as `LanguageHome.EN_US`, write the resolved concrete value in the default/value context and append sibling constants or enum members to the field `說明` as possible concrete values when available; do not render `CONSTANT=value` mappings, and exclude QS metadata constants such as `CACHE_REGION` and `UNIT_ID` from possible values. Do not chase constants into dependency jars or build a full Java type index/classpath just to infer one field; leave those as `unknown` when source is not available.
- Keep fast-export memory bounded: do not retain full Java source, large method bodies, or the complete Markdown report longer than needed. Compact API records before rendering, clear runtime caches after discovery, and stream large reports to disk when possible.
- Inspect each module's QS unit SQL metadata, `init.sql`, `init.xml` SQL initialization baseline, Gradle/module files, resource configuration, and fallback Java entry points before writing the API Markdown report. Apply SQL initialization baselines per module project: each `*-module-*` uses only its own `src/main/resources/QS-MODULE/data/sql/default/init.xml`, and versions from one module never affect another module. When a module's `init.sql` exists without `init.xml`, treat that module's `init.sql` as containing all SQL and scan only `init.sql`. When that module's `init.xml` has a `<version>`, treat its `init.sql` as containing that version and earlier incremental SQL; scan that module's `init.sql` plus only versioned `.sql` files with a filename version greater than the module's own baseline.
- Scan all `.sql` files under bundled QS base unit metadata directory `assets/templates/api-discovery/sql/` before module SQL, then merge each module's own SQL metadata on top. The template SQL files provide compact QS product metadata, primarily `TsUnit` rows and any focused field metadata needed for API discovery. If an application API class extends a QS product API class, bind it through the superclass `FApiClassName`, for example ECP `UserApiImpl` extending `com.jeedsoft.quicksilver.user.api.impl.UserApiImpl` belongs to `Qs.User`; do not require the application module SQL to contain the QS product unit.
- If SQL metadata, template SQL, and resolvable superclass source still cannot identify the unit, infer the unit from the `extends` superclass FQCN before using the current subclass. `com.jeedsoft.quicksilver...` implies product prefix `Qs`; otherwise skip common domain/organization prefixes such as `com.xxx`, `org.xxx`, `net.xxx`, or `cn.xxx`, and use the segment after the second dot as the product prefix with initial uppercase. For example, `com.jeedsoft.quicksilver.user.api.impl.UserApiImpl` implies `Qs.User` / `qs/user`, `com.chainsea.abc.order.OrderApiImpl` implies `Abc.Order` / `abc/order`, and `org.foo.mes.order.OrderApiImpl` implies `Mes.Order` / `mes/order`.
- Derive `<project-name>` from the discovered QS child project names first, such as `<project-name>-module-main`, `<project-name>-run`, `<project-name>-build`, or `<project-name>-lib-xxx`, by taking their shared prefix. Fall back to the QS parent directory basename only when no reliable shared prefix can be derived.
- Record findings in Markdown under the QS parent project's `docs/` directory, defaulting to `docs/<project-name>-api-discovery.md`, such as `docs/quicksilver-api-discovery.md`.
- Write the output as an AI-friendly API calling reference for frontend development: start with prominent `# 報告摘要` using Traditional Chinese labels such as `專案根目錄`, `掃描模組`, `API 總數`, and `格式`; `專案根目錄` must show the logical project name, not a local absolute filesystem path, and `格式` should be `路徑優先精簡 API 參考`. Then write `# 完整 URL 拼接` as multiple readable Markdown lines, not one compressed paragraph. After the URL section, write `# Token API` first for token-operation routes, then `# 一般 API` for all remaining routes; each API must appear exactly once. Under each section, write each API as an HTTP-method-and-path-led block such as ``## `POST openapi/qs/user/token/apply` `` or ``## `GET openapi/qs/file/download/{path}` ``. Separate adjacent API blocks with `---`.
  Infer the API purpose using explicit developer text first: `@Api.description` is first priority; if absent, the first non-empty line of the method's directly attached JavaDoc, block comment, or consecutive line comments is second priority. A method comment may appear immediately before `@Api` or between `@Api` and the method signature. Ignore decorative separator comments and paired separator group blocks, such as `// ----------` / `// for BusinessEntityApi` / `// ----------`; the middle line is a group label, not an API purpose. Ignore comment lines that start with `@`, such as `@deprecated use EntityApiImpl`, and continue looking for a non-`@` line. If neither exists, infer a best-effort short description from final API path: identify action words such as token/upload/download/transfer/status/login/logout/password/import/export, translate only the needed business object words with a small domain glossary, and render them through natural Traditional Chinese action templates. When working interactively and an unknown business word cannot be translated from code, metadata, or the local glossary, it is acceptable to search the web for the term's public technical/business meaning, then use the confirmed meaning conservatively and prefer adding durable translations back into the glossary or script rules. Then use explicit handler behavior, `@Api.value` / `@Api.path`, unit name, return type, and request/response construction as supporting evidence. If a precise business purpose cannot be inferred from route or metadata, fall back to translating the current `@Api` handler method name into Traditional Chinese, including both verb-first names such as `updateMessage` and object-plus-verb names such as `agentIsWorkTime`. Do not output untranslated or camelCase fallback phrases such as `處理downloadfax`; keep the purpose conservative or omit it when no natural translation is possible. For non-entity API classes or units, identified by Java inheritance or by `TsUnit.FEditId` edit types documented in `references/unit-core.md`, do not use the unit name as the business target in purpose text. Write the purpose as a short unlabeled paragraph directly below the API heading and before `基本資訊`; every line of this paragraph must be indented by two full-width spaces, such as `　　新增商務訂單。`. Use stable Traditional Chinese call-detail labels like `**基本資訊**`, `**路徑參數**`, `**請求參數**`, `**請求標頭**`, `**請求內容**`, and `**回應內容**` as applicable, and keep all request/response details inside that API block so AI can identify every API without chasing shared catalogs. In each API's `基本資訊`, include `項目名稱` with the QS module project name that provides the API class, such as `ecp-module-main`, `HTTP 方法` with `GET` or `POST`, and `起始版本` when `@Api.since` is non-empty so parent projects with multiple module projects and API versions can distinguish API ownership and availability. Field analysis is code-first: use handler code as the source of truth and `@Api.input` / `@Api.output` only as fallback or supplemental metadata. In field tables, render annotation-only source as `API 註解`, not raw `@Api.input` or `@Api.output`; combined source should read `處理器程式碼 / API 註解`. Choose heading levels or bold labels that keep the context readable; do not force every API path or detail block to a fixed heading level. Do not add `源碼`, `源碼依據`, or source-evidence rows to `基本資訊`; source evidence is for validation and field-level `來源` columns only. Do not output grouped top-level catalog sections such as `Scope`, `Summary`, `Shared EntityApiImpl Contract`, `Unit API Bindings`, or `Exported Routes` unless the user explicitly asks for an audit/catalog-style report.
- Use source file evidence to validate every confirmed API finding, but do not render source evidence as a `基本資訊` row. Mark inferred values explicitly inside the relevant API section instead of guessing.

## Template Layout

- `assets/templates/create-qs-project/parent/`: templates for the QS parent project root.
- `assets/templates/create-qs-project/build/`: templates for `<project-name>-build/`.
- `assets/templates/create-qs-project/run/`: templates for `<project-name>-run/`.
- `assets/templates/create-qs-project/lib/`: full template tree for `<project-name>-lib-<subproject-name>/`.
- `assets/templates/create-qs-project/module/`: full template tree for `<project-name>-module-<subproject-name>/`, including web resources when present.
- `assets/templates/unit/java/`: Java templates for unit model, DAO, service, action, API, and home layers.
- `assets/templates/unit/sql/`: SQL templates for unit metadata and table creation.
- `assets/templates/api-discovery/java/`: API annotation contract reference used when analyzing QS module APIs.
